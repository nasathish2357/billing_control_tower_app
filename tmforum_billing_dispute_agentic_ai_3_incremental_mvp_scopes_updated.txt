Updated Incremental MVP Roadmap: TM Forum Catalyst Telecom Billing Dispute Agentic AI Backend
===========================================================================================

Purpose
-------

This document updates the 3-phase incremental MVP roadmap based on technical implementation feedback.

The roadmap still splits the original full backend design into 3 incremental MVP scopes, but it corrects several important implementation risks around LangGraph, LangChain, KG timing, LLM provider strategy, streaming chat, and human-in-the-loop state resumption.

Key updates from prior roadmap:

- Removed the restriction to use a LangChain agent factory API.
- Removed Azure OpenAI switching requirement.
- Kept only Google Vertex AI Gemini as the LLM provider.
- Added streamable chat API requirement using Server-Sent Events, SSE.
- Corrected LangGraph architecture to avoid AgentExecutor nested loops.
- Moved NetworkX/kglab KG out of MVP 1 and into MVP 2/MVP 3.
- Added shallow tool schema requirement for Gemini compatibility.
- Added explicit LangGraph checkpointer and interrupt/resume handling for human approval flows.

Non-negotiables across all MVPs:

- FastAPI backend.
- PostgreSQL persistence.
- LangGraph orchestration using StateGraph.
- Google Vertex AI Gemini as the only LLM provider.
- LangChain-compatible chat model and tool binding where useful.
- Use bind_tools-style tool binding, not AgentExecutor or hidden agent loops inside graph nodes.
- Owner-layer orchestration boundaries.
- Audit trace for important actions.
- TMF-inspired mock APIs.
- Streamable chat API for demo UX.
- 10 use cases remain visible from MVP 1 onward.
- Architecture rule remains: Write-local, read-federated, orchestrate-through-owner.


Critical Technical Corrections
------------------------------

1. LangGraph owns the agentic loop
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Do not use LangChain AgentExecutor or hidden prebuilt agent loops as a mandatory pattern inside LangGraph nodes.

Correct pattern:

- LangGraph StateGraph owns orchestration.
- LLM nodes call Gemini with bound tools when reasoning/tool selection is needed.
- Tool execution is routed through explicit LangGraph tool nodes or deterministic service nodes.
- Graph state is visible, inspectable, checkpointed, and resumable.

Avoid:

- Hidden while loops inside AgentExecutor.
- Nested agent loops inside LangGraph nodes.
- Unbounded recursive tool calling.
- Large all-powerful agents with unrestricted tools.

Recommended pattern:

LLM node:

- Receives graph state.
- Produces response or tool call request.
- Uses shallow, strictly typed tool schemas.

Tool node:

- Executes approved tool/service.
- Writes result back to graph state.
- Emits audit event if required.

Router node:

- Decides next node based on state, policy, and tool results.


2. Gemini-only LLM provider
~~~~~~~~~~~~~~~~~~~~~~~~~~~

The MVP roadmap now uses only Google Vertex AI Gemini.

Environment variable:

LLM_PROVIDER=vertexai_gemini

Provider responsibilities:

- Read Vertex AI project, location, and model configuration.
- Return configured Gemini chat model.
- Return configured Gemini-compatible embedding model if used.
- Fail fast with a clear error if required config is missing.

Suggested files:

app/llm/provider.py
app/llm/gemini_provider.py

Remove Azure OpenAI requirements from implementation scope.


3. Gemini-compatible shallow tool schemas
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Tool schemas must be shallow and strictly typed.

Do not expose deeply nested TMF payloads directly as tool schemas.

Rules:

- Tool input schemas should be small Pydantic models.
- Prefer IDs and simple filters over full nested objects.
- Return complex TMF-style data from tools as controlled JSON payloads, not as deeply nested tool input schemas.
- Avoid recursive schemas.
- Avoid highly polymorphic unions.
- Avoid huge optional field sets.
- Avoid passing full TMF Swagger-style objects into LLM tool definitions.

Good tool input example:

{
  "billing_account_id": "BA-100001",
  "bill_id": "BILL-202604"
}

Bad tool input example:

- Full TMF CustomerBill object with nested relatedParty, relatedEntity, billItem, product, taxItem, characteristic, adjustment, payment, and recursively nested references.

Tool design principle:

The LLM selects intent and calls small tools. Backend services retrieve and process complex telecom payloads internally.


4. Streamable chat API
~~~~~~~~~~~~~~~~~~~~~~

The chat API must support streaming for demo quality and latency handling.

Primary chat endpoint:

POST /api/v1/chat/stream

Recommended transport:

- Server-Sent Events, SSE.

The stream should emit events such as:

- conversation_started
- intent_detected
- tool_started
- tool_completed
- evidence_loaded
- token
- remediation_started
- remediation_completed
- escalation_created
- audit_written
- final
- error

Optional non-streaming endpoint may remain for automated tests:

POST /api/v1/chat

But the demo-facing endpoint should be /api/v1/chat/stream.


5. KG timing correction
~~~~~~~~~~~~~~~~~~~~~~~

Do not implement NetworkX/kglab in MVP 1.

In MVP 1, use PostgreSQL relational queries and JSONB payloads for lightweight context reasoning.

For example:

Customer -> BillingAccount -> Bill -> BillLineItem -> UsageRecord

This can be handled using SQL joins and structured JSONB evidence without maintaining a separate in-memory graph.

Introduce NetworkX/kglab in MVP 2 or MVP 3 after the data model and flow behavior are stable.


6. Human-in-the-loop resumption with LangGraph checkpointer
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

Human approval flows must explicitly use LangGraph checkpointing.

MVP 2 and MVP 3 approval flows must:

- Run graphs with a stable thread_id.
- Use PostgreSQL checkpointer, for example PostgresSaver.
- Interrupt before or after high-risk action nodes.
- Persist graph state before returning pending approval.
- Resume graph execution using the same thread_id after approval/rejection.

High-level flow:

1. Graph reaches high-risk action.
2. Policy node marks approval_required=true.
3. Graph interrupts and checkpoint is saved.
4. API returns approval_id and thread_id.
5. Human approves or rejects.
6. Resume API reloads graph state using thread_id.
7. Graph continues to execute or gracefully closes as rejected.

Do not keep approval state only in memory.
Do not block a web request waiting for human approval.


Overall Incremental Strategy
----------------------------

MVP 1: Foundation and first executable flows

- Foundation backend and first demo-ready flows.
- Implements P2, P3, and P8 as full executable flows.
- Keeps remaining 7 use cases deterministic.
- Establishes FastAPI, PostgreSQL, LangGraph, Gemini, RAG/playbook retrieval, audit, streamable chat, and demo APIs.
- Uses SQL/JSONB context reasoning instead of NetworkX/kglab.

MVP 2: Expanded cross-domain assurance and governance

- Converts more deterministic use cases into full LangGraph flows.
- Adds L1 Monitoring Agent, L1 Closed-Loop Control Agent, and stronger audit governance behavior.
- Adds event correlation, approval flow with LangGraph checkpoint interrupt/resume, service problem lifecycle, and richer BSS/OSS/network mock evidence.
- Introduces NetworkX/kglab KG only after MVP 1 is stable.
- Implements P1, P4, P5, P6, and P9 as full flows.

MVP 3: Complete original scope and catalyst-grade backend

- Converts all remaining use cases to full flows.
- Implements complete layer sub-agent ecosystem.
- Expands chat/self-service capabilities.
- Adds stronger KG/RAG, simulation governance, and broader TMF-inspired APIs.
- Implements P7 and P10 fully and hardens all 10 flows.

MVP 2 New Full Flow Details
---------------------------

P1. Missing or delayed usage feed into billing

MVP 2 implementation:

- L4 publishes suspected usage delay event.
- Event DB stores event.
- L1 Monitoring component correlates event.
- L1 Master creates control case.
- L1 requests L3 mediation backlog/replay evidence.
- L1 requests L2 bill-run exposure and account impact evidence.
- L1 Closed-Loop Control decides whether replay and bill protection are allowed.
- L3 executes mediation replay mock action.
- L2 applies bill protection or affected charge hold.
- Audit trace records event, evidence, decisions, and actions.

P4. QoS or slice entitlement mismatch with premium charging

MVP 2 implementation:

- L2 identifies premium charging dispute or pre-bill exposure.
- L1 requests L3 service QoS evidence.
- L1 requests L4 slice/network performance evidence.
- L2 Product & Entitlement tool validates promised commercial tier.
- L1 evaluates delivered-vs-promised mismatch.
- L2 removes premium surcharge or applies correction.
- Audit trace stores entitlement, service, and network evidence.

P5. Service degradation requiring commercial compensation

MVP 2 implementation:

- Customer/care raises Trouble Ticket.
- L2 triages whether technical evidence is required.
- L2 escalates to L1 only when service evidence is needed.
- L1 requests L3 service degradation/SLA evidence.
- L1 decision node determines compensation recommendation.
- L2 applies compensation or creates human approval request if threshold exceeded.
- If approval is required, LangGraph interrupts and checkpoints state.
- Audit trace stores service evidence, compensation decision, and approval state.

P6. Collections hold required during billing dispute

MVP 2 implementation:

- Customer raises invoice dispute.
- L2 Trouble Ticket tool opens/updates ticket.
- L2 Collections tool checks collections/dunning state.
- L2 applies disputed balance hold.
- L2 separates disputed from undisputed debt.
- Audit trace stores policy basis and hold action.

Note:

- P6 normally remains L2-only.
- L1 should not be involved unless cross-domain evidence is required.

P9. Internal anomaly discovered by monitoring before customer impact

MVP 2 implementation:

- Origin layer publishes BillingRiskDetectedEvent.
- L1 Monitoring component correlates event with KG/RAG patterns.
- L1 Master creates proactive control case.
- L1 requests focused evidence from relevant layer orchestrators.
- L1 Closed-Loop Control decides preventive action.
- L2/L3/L4 execute only their own actions.
- Audit trace stores proactive prevention before customer impact.


MVP 2 KG Introduction
---------------------

Introduce NetworkX/kglab in MVP 2 only after MVP 1 is stable.

KG should be rebuilt or synchronized from PostgreSQL seed/state data using controlled scripts or service methods.

Do not rely on ad-hoc in-memory graph mutations without persistence strategy.

Recommended approach:

- PostgreSQL remains source of truth.
- KG is a derived read model.
- Build/rebuild graph from DB records.
- Store graph snapshot metadata.
- Use KG for relationship queries and reasoning, not transactional writes.

MVP 2 KG relationships:

- Customer -> BillingAccount -> Bill -> BillLineItem.
- Customer -> Product -> Offer -> Plan -> AddOn -> Promotion.
- Product -> Service -> Resource -> NetworkFunction.
- Service -> QoSProfile -> SliceProfile.
- UsageEvent -> MediationRecord -> RatedUsage -> BillLineItem.
- TroubleTicket -> ControlCase -> EvidencePack -> RemediationAction.
- ServiceProblem -> AffectedService -> AffectedResource.
- Event -> ControlCase -> Issue.

MVP 2 KG queries should support:

- Find impacted customer/account from issue.
- Find service/resource path from billing account.
- Find bill line item impacted by usage/session.
- Find related trouble ticket/service problem for control case.
- Find prior similar issue pattern.

MVP 3 Full P7 Flow
------------------

P7. Partner or wholesale charge discrepancy

Implementation:

- L2 opens partner-charge/roaming Trouble Ticket.
- L2 determines whether partner/network evidence is needed.
- L2 escalates to L1 when cross-domain/partner evidence is needed.
- L1 requests L4 roaming/exposure evidence.
- L4 Roaming / Exposure component checks roaming context, PLMN/partner event anomaly, SEPP/NEF/API exposure evidence if applicable.
- L1 requests L2 retail bill/customer impact evidence.
- L2 Roaming & Partner Settlement component separates retail customer outcome from downstream settlement recovery.
- L2 applies retail correction if customer was incorrectly charged.
- L2 creates partner settlement recovery case.
- Audit trace links customer outcome to partner recovery process.

Expected outcome:

- Customer issue is resolved in L2.
- Partner settlement case is created separately.
- L4 provides evidence but does not perform financial actions.
- Audit trace preserves retail-to-wholesale evidence chain.


MVP 3 Full P10 Flow
-------------------

P10. Governed end-to-end dispute handling across BSS, OSS, and network teams

Implementation:

- Entry layer raises cross-domain issue through event/escalation trigger.
- L1 Monitoring component correlates and prioritizes.
- L1 Master creates governed control case.
- L1 requests BSS evidence/actions through L2 BSS Orchestrator.
- L1 requests service evidence/actions through L3 OSS Orchestrator.
- L1 requests network evidence/actions through L4 Network Orchestrator.
- L1 Closed-Loop Control evaluates risk and policy.
- L1 Human-in-the-Loop Approval component pauses high-risk actions using LangGraph interrupt/checkpointer.
- Each layer executes only its own approved actions.
- L1 Audit & Compliance component records evidence, approvals, payload traces, and outcomes.
- Control case closes only when each domain outcome is synchronized.

Expected outcome:

- Multi-team dispute is governed centrally but executed locally.
- Owner-layer boundaries are preserved.
- Full evidence/action/approval chain is auditable.

Recommended Implementation Discipline
-------------------------------------

For every MVP:

- Keep code simple and testable.
- Do not over-engineer beyond the MVP stage.
- Use Gemini-only provider unless future requirements change.
- Do not use AgentExecutor or hidden prebuilt agent loops as a mandatory LangGraph node pattern.
- Use LangGraph StateGraph to own agent loops.
- Use bind_tools-style tool binding and explicit tool nodes where LLM tool calling is needed.
- Keep tool schemas shallow and strictly typed.
- Do not bypass owner-layer orchestration.
- Do not claim functionality works unless tests pass.
- Keep deterministic fallbacks where LLM calls are unavailable.
- Preserve auditability for every important action.
- Use JSONB pragmatically for complex telecom mock payloads.
- Keep TMF APIs clearly marked as mock/TMF-inspired until real conformance work is planned.
- Use streaming for chat demos to avoid perceived latency and UI freezes.
