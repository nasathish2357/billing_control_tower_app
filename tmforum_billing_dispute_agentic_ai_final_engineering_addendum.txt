Final Engineering Addendum: Coding-Phase Micro-Optimizations
================================================================

This addendum should be applied to the updated 3-incremental-MVP roadmap for the TM Forum Catalyst Telecom Billing Dispute Agentic AI backend.

The roadmap has no remaining critical correctness issues. These are final implementation guardrails for coding agents and engineering teams.


1. Keep LangGraph State Lightweight
-----------------------------------

LangGraph checkpointing persists graph state at graph steps. Therefore, the graph state must remain small and reference-oriented.

Coding rule:

- Do not store large TMF-inspired JSON payloads directly inside LangGraph state.
- Do not append full bills, full event histories, full audit payloads, or large evidence records into graph state.
- Store large payloads in normal PostgreSQL tables such as evidence_packs, audit_traces, events, bills, usage_records, or JSONB evidence columns.
- Put only compact references and summaries in LangGraph state.

Recommended state fields:

- issue_id
- control_case_id
- thread_id
- evidence_pack_ids
- audit_trace_id
- small evidence_summary
- policy_decision
- planned_action_ids
- final_response_summary

Avoid state fields like:

- full_customer_bill_payload
- full_tmf_payload
- full_usage_event_list
- full_network_trace
- full_chat_transcript_with_tool_payloads

Recommended pattern:

Tool retrieves or creates large evidence payload.
Tool stores payload in PostgreSQL evidence_packs table.
Tool returns evidence_pack_id plus short summary.
LangGraph state stores only evidence_pack_id and summary.

Example lightweight state update:

{
  "issue_id": "ISSUE-P3-001",
  "control_case_id": "CC-P3-001",
  "evidence_pack_ids": ["EVP-P3-L4-001", "EVP-P3-L2-001"],
  "evidence_summary": "L4 confirmed missing session release; L2 confirmed impacted charge item.",
  "policy_decision": "auto_remediation_allowed"
}


2. Use Async FastAPI, Async SQLAlchemy, and Async LangGraph Execution
--------------------------------------------------------------------

The backend should be async-first to avoid blocking FastAPI's event loop, especially for streaming chat.

Coding rule:

- Use FastAPI async route handlers.
- Use SQLAlchemy async engine and AsyncSession.
- Use async database repository methods.
- Use LangGraph async execution where available.
- Use async streaming for /api/v1/chat/stream.
- Avoid blocking graph.invoke() in request handlers when an async alternative exists.

Recommended patterns:

- Use SQLAlchemy ext.asyncio.
- Use async_sessionmaker.
- Use AsyncSession in dependencies.
- Use await graph.ainvoke(...) for non-streaming graph execution.
- Use await graph.astream_events(...) or async iteration over graph streaming events for SSE.
- Use AsyncPostgresSaver or async-compatible LangGraph Postgres checkpointer when available.

For streamable chat:

- The /api/v1/chat/stream endpoint should return StreamingResponse.
- It should asynchronously iterate over LangGraph stream events.
- It should emit SSE event frames for tool status, tokens, final response, and errors.

SSE event style:

conversation_started
tentative_intent_detected
tool_started
tool_completed
evidence_loaded
token
audit_written
final
error

Avoid:

- Blocking database sessions inside async endpoints.
- Long synchronous graph.invoke() calls in streaming endpoints.
- Large synchronous file reads during chat response streaming.
- CPU-heavy operations directly inside the event loop.

If a CPU-heavy operation is unavoidable, move it to a background worker or threadpool.


3. Add Google Vertex AI Local Authentication Configuration
---------------------------------------------------------

The MVP uses Google Vertex AI Gemini only. Local and Docker-based development must make authentication explicit.

Coding rule:

- Add GOOGLE_APPLICATION_CREDENTIALS to .env.example.
- Document how the local service account file is mounted into Docker.
- Fail fast with a clear error if Vertex AI credentials are missing.

Required .env.example addition:

GOOGLE_APPLICATION_CREDENTIALS=/app/secrets/vertex-ai-service-account.json

Recommended Docker/local development note:

- Mount the local service account JSON file into the container at /app/secrets/vertex-ai-service-account.json.
- Do not commit service account JSON files.
- Add secrets/ and credential JSON paths to .gitignore and .dockerignore as appropriate.

Example docker run mount pattern:

-v /local/path/service-account.json:/app/secrets/vertex-ai-service-account.json:ro

Provider validation should check:

- VERTEXAI_PROJECT is set.
- VERTEXAI_LOCATION is set.
- VERTEXAI_MODEL is set.
- GOOGLE_APPLICATION_CREDENTIALS points to an accessible file in local/Docker execution, unless the runtime environment uses workload identity or another Google-supported credential mechanism.

Updated Engineering Discipline
------------------------------

In addition to the existing implementation discipline, coding agents must follow these rules:

- Keep LangGraph state small and reference-based.
- Store large evidence and TMF-style payloads in PostgreSQL, not graph state.
- Prefer async FastAPI routes, async SQLAlchemy sessions, and async LangGraph execution.
- Use streaming-first chat implementation for demo UX.
- Make Vertex AI authentication explicit in local and Docker setup.
- Never commit Google service account credential files.
